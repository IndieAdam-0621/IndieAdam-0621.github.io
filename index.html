Asset References

AssetReference 是一个可以用来引用可寻址化资源的类型。

AssetReference类可以在MonoBehaviour和ScriptableObject中使用，当为MonoBehaviour对象或者ScriptableObject对象添加一个可序列化的filed时，我们就可以在这个对象的Inspector面板上去为这个field赋值，我们可以通过类型和标签来限制field的赋值类型。

上图中的各种资源引用（AssetReference可以直接用 Addressable Asset 赋值，把需要引用的资源拖拽到对应的field中去即可）

如果拖拽了一个没有设置为可寻址化的资源到这里的field中，Unity将会自动把这个资源变成可寻址资源并将它添加到默认的addressables group。

⚠注意：
如果要将一个Addressables Group中的资源分配到 AssetReference field 中，那么这个组的 高级选项中的Include GUID in Catalog 选项必须要勾选上。Include GUID in Catalog 选项默认是勾选的。

Addressables系统总览

Overview of the Addressables system

可寻址化为我们提供了一个可以与项目共同发展共同成长的系统。

本文设计的专有名词及其概念：

  1. Asset address: 标记可寻址资源的一个字符串id，我们可以通过一个地址来加载我们想要的资源
  2. AssetReferences:
  3. Label: 可以分配给一类资源的标签，通过这个标签可以加载与这个标签相关的所有资源
  4. Asset location: 一个运行时的object，

⭐默认情况下，Addressables是通过AssetBundles来打包我们项目中的资源的，不过我们也可以通过 自定义一个实现IResourceProvider 接口的类来通过其他方式操作和管理我们的资源。

Asset addresses 资源地址（索引）

可寻址系统的一个重要特点就是我们会为资源分配地址，并通过这些地址在运行时加载这些资源。

可寻址化资源管理器会在内容目录(content catalog)中寻找你指定的地址，并由此找到所要加载的资源的位置。（这些资源可以是构建在应用内的(build-in application)，资源管理器会先加载你要加载的资源的依赖项，如果有需要的话还会从远端下载。

由于这个地址与资源的实际存储的位置并不是绑定的关系，这就为管理和操作资源提供了更高的灵活性（不管是在Editor中还是在运行时），地址(address)与资源的实际存储位置是由Content catalogs映射的。

我们的资源应该是只有唯一的一个地址，但是这个地址可以是不唯一的（可以分配给很多不同的资源），比方说我们的一个资源有很多变种（蓝色T恤，白色T恤。。。），那这些资源可以是同一个adress，但是可以额外通过标签去区分它们：

  • Assets 1: address: “t_shirt_texture”, label:”blue”
  • Assets 2: address: “t_shirt_texture”, label:”white”

可寻址系统的API方法中有些是只会加载一个资源的，比如 LoadAssetAsync，它只会加载你指定的address对应的一个资源，如果说你把这个address分配给了好多个资源的话，这个方法会返回它找到的这些资源中的第一个。当然，也有一口气加载好多资源的接口，比如LoadAssetsAsync，如果你把一个address分配给了好多不同的资源，那么这个接口将会把所有分配了这个address的资源全部给你找出来。

ℹ Tip:
可以通过LoadAssetsAsync方法的MergeMode参数来加载两个指定条件的交集，按上面的例子来说就是，我们可以把address和label两个key作为查找资源的条件。

AssetReference 资源引用

AssetReference就是一个可以被任何设置为Addressable的资源赋值的类型，我们只需要加载我们想要的那个“主资源”，跟这个“主资源“相关的（该资源所依赖的）其他资源Addressable会自动加载。

通过AssetReference就可以实现在MonoBehaviour脚本或者ScriptableObject脚本中定义一个可以赋值的域(field)，这样就可以很方便地在Inspector面板中将想要设置地资源直接拖拽赋值，可以用来代替为资源分配和定义地址(address)的方式。

除了最基本的 AssetReference 类型，Addressables还提供了一些特定的类型，比如：AssetReferenceGameObject 和 AssetReferenceTexture，通过他们可以限定一些资源的类型，避免在拖拽赋值到field的时候产生的类型错误。除此之外，我们还可以通过一个属性 AssetReferenceUILabelRestriction来限制域中赋值的资源的标签必须为特定标签。

Loading and releasing assets 加载与释放资源

通过 address 或者 label 或者 AssetReference 都可以加载你想要的资源。

如果一个资源在运行时不需要再操作和使用了就必须要释放掉，否则会一直占用内存，Addressable系统对已经加载的资源会维护一个引用计数，如果这个计数归零了，Addressable系统才会自动去回收（释放）这个资源（以及对应的内存），所以我们不需要追踪记录一个资源的引用情况，我们只需要保证我们每一个加载引用的具体的资源，当我们不需要再使用的时候，把他们释放掉，就好了。

比方说，当你卸载一个场景资源，场景中那些隐含的资源（隐含的资源是指啥？）是不会自动卸载的，必须通过 Resources.UnloadUnusedAssets 或者 UnloadAsset 去释放这些资源。注：如果你是在 LoadSceneMode.Single 模式（当加载一个场景的时候会关闭其他所有场景）下去加载场景的，那么Unity是会自动在你卸载场景的时候调用 UnloadUnusedAssets 去卸载其他不再使用的资源的。

可寻址资源管理系统

Get Started With Addressables

[TOC]

Session Goals 本篇文章的概述及目标

你将在本篇文章中，获得以下信息

  • 为什么我们的Unity项目需要资源管理系统
  • 如果开始上手Unity中最新的资源管理方式：Addressables
  • 可寻址资源与之前的资源管理系统对比是怎样的
  • 在项目中使用可寻址资源的好处
  • 可寻址资源与对象池的区别

参考视频链接: Getting started with Addressables | Unite Now 2020

参考文档链接: Unity Addressable Asset System

Conception 基本概念

Addressables 可寻址资源系统提供了一套用来组织和打包游戏资源的工具，以及在运行时加载和释放这些资源的API

当一个asset被设置为Addressable之后，我们就可以通过这个资源的地址在任何地方去加载和使用这个资源，不论这个资源是在本地还是在云端，Addressable系统会定位找到指定的资源并将它们返回给我们来使用。

Advantage 可寻址资源系统的优势及特点

可寻址资源系统将会在以下几个方面改进我们的项目：

  • 灵活性：可以自由配置项目中资源的位置，比如你可以直接放在包体里面，直接让玩家随游戏一起完整下载；或者你也可以把部分资源单独放在云端，等玩家需要用到的时候再去下载。另外假如要修改资源的地址，也不需要修改代码，只需要更改相关的配置就行了，随时都能更改，无需牵动代码。(You can change where you access a specific asset at any stage in your project without rewriting any game code.)
  • 依赖管理：当你加载一个资源的时候，系统会自动加载跟这个资源相关的所有依赖项，以便在系统将内容返回之前加载所有网格、着色器、动画和其他资产。(The system automatically loads all dependencies of any assets you load, so that all meshes, shaders, animations, and other assets load before the system returns the content to you.)
  • 内存管理：Addressables系统会帮助我们加载资源，同时也会帮助我们卸载不再需要的资源，它会自动进行引用计数，从而及时地做内存的清理和维护。并且Addressables系统还提供了一个非常成熟且健壮的profiler工具
  • 资源打包：即使移动资源的位置或者改变资源的命名，它也能够通过映射和复杂的依赖关系链来有效率地自动打包AssetBundles。我们可以将一部分资源放到云端，在安装好应用/游戏之后再开始下载，以此来减少初始包体大小。

⭐Note: Addressables 系统是基于 AssetBundles 去构建的，如果我们想要在项目中使用AssetBundles但是又不想写详细的管理代码，那么就应该使用 Addressables 系统来管理我们的资源.

给现有项目添加Addressables系统

如果要将现有的项目应用Addressables系统来管理资源，只需要通过包管理器去安装一个Addressables package（下文有安装方法）就可以了，安装完成后需要给所有的资源指派他们的“地址（addresses)”并且需要重构相关的运行时的资源加载卸载调用等代码。（下方的升级到Addressabls章节有更多详细的信息）

虽然可以在项目的任何阶段开始应用Addressables系统，但是肯定还是最开始的时候就用比较好，避免后期代码重构和资源重新设置的一些工作时间上的消耗。

Start 开始使用

总体的步骤有三个：

  1. 把资源都置为可寻址资源
  2. 通过Addressable的API引用并加载这些资源
  3. 构建可寻址资源

可参考案例：

See the Space Shooter project in the Addressables-Sample repository for an example of a project set up to use Addressable assets.

安装与设置Adressable的流程：

  1. Installing the Addressable Assets package at package manager

  2. Marking Assets as addressable 将资源标记为可寻址资源

    There are several ways to mark an item as an Addressable Asset:

    1. Select an asset and click the Addressable check box in the Inspector window.

    2. 初始化 Addressable Group 设置，点击 Create Addressables Settings

      点击 Create Addressables Settings 之后，会生成一个AdressableAssetData的文件夹，用来存储设置文件和用于跟踪Addressables设置的其他资源。

    3. 把资源放到一个设置了Addressable的文件夹中

  3. The default address for our asset is the path to the Asset in our project. For example Assets/Prefabs/Sphere.prefab.

⭐如果将Resources文件夹里面的一个资源文件设置为Adressable的话，Unity会把这个文件从Resources文件夹中移出，设置为Addressable的资源可以放在除了Resources文件夹以外的任何文件夹。

使用标记为Adressable资源的流程：

要加载一个Addressable资源大体可以通过以下方式：

  • 用一个引用资源的AssetReference
  • 用它的地址字符串
  • 使用分配给资源的标签
（一）通过资源引用 AssetReference
（二）通过地址加载

（加载单个资源）通过地址字符串加载资源，要记得用完了还要释放掉它。

（三）通过标签加载

（加载多个资源）可以通过标签加载标签相同的一批资源

管理可寻址资源

通过 Addressables Group 窗口去管理可寻址化资源，在该窗口中可以创建可寻址化资源组：Addressable Group，可以将可寻址化资源在这些组之间移动，给这些资源分配地址和标签。

最开始的时候这里面有一个默认的组，设置为可寻址化的资源都会自动放到这个默认组里，但是为了后期方便管理（资源分类、加载、内存等等），最好是添加不同的分组，给资源分类放置，具体根据项目情况决定。

资源组的关键设置（右键单击一个资源组，点击 Inspect Group Settings）：

  • 构建路径 Build Path：资源内容构建完成后，将这些内容存储到哪里
  • 加载路径 Load Path：游戏在运行时加载构建完成后的资源的路径
  • 包模式 Bundle Mode：资源打包的方式
    • Pack Together 将该资源组中的所有资源打进一个包里
    • Pack Separate 资源组中的每个资源单独打包 （当我们将一个文件夹设置为 Addressable 并且想要所有内容构建在一起时特别有用）
    • Pack Together By Label 将标签相同的资源打在一个包里
  • 内容更新限制 Content update restriction：开启这个选项的话后期可以发布一些比较小的更新，可以构建内容更新包，如果你要发布的游戏不需要从远程下载更新或者额外内容，后期也不用热更，那这个设置就可以不管了。

构建可寻址化资源

可寻址化资源的构建过程会基于之前的资源组设置和Editor在当前平台的设置将我们安置在可寻址资源分组(Addressables groups)中的资源转化为 AssetBundles。

在Unity 2021.2以上的版本中，我们可以将可寻址资源的构建作为游戏或应用构建的一部分去处理，或者也可以将二者分开，在构建整体游戏或应用前先构建可寻址化资源。如果设置了让播放器（打包出的游戏或应用程序）和可寻址化资源构建一体化的话，当通过基本的Build功能或Build and Run功能去构建应用的时候，Unity会在真正构建应用前先构建可寻址化资源（Pre-Build)。

在Unity早期版本中，如果要分开构建资源和应用，需要在 Addressable Group 窗口 的 Build 菜单中去手动点击构建：

之后如果构建应用的话，Unity会使用最近构建过的Addressable Assets针对当前平台进行构建。我们也可以自定义构建脚本来实现可寻址资源构建自动化。

关于上图 Build 菜单中的几个选项：

  • New Build 使用特定的脚本来进行构建：如果没有自定义脚本的话，就是用Unity默认的脚本进行构建。
  • Update a Previous Build 更新之前的一个构建结果：要更新之前的一个构建，可寻址化系统需要之前构建所产生的 addressables_content_state.bin 文件，这个文件被放在项目的 Assets/AddressableAssetsData/Platform 下。
  • Clean Build 删除之前构建的缓存

默认情况下，创建出来的文件会放置到配置文件中所设置的 LocalBuildPath 和 RemoteBuildPath 中，这些文件包含了Unity构建应用时所需要的 AssetBundles(.bundle), catalog JSON, hash files, setting files.

⚠注意
在大多数情况下，不应该修改默认的本地构建路径或本地加载路径，如果修改了的话，则必须在构建游戏或应用前将自定义路径下的资源文件手动复制到项目的 StreamingAssets 文件夹中。并且需要注意的是，如果设置了资源应用打包一体化的话（就是在构建应用的同时构建资源），这些自定义的路径也不会包含在其中！！！

如果之前的Build Path设置的是远程路径，那还需要手动上传所有的 AssetBundles，catalog以及hash files到服务器主机上。如果项目用不到远程下载资源的功能，那就把构建路径和加载路径都设置为本地就可以。

构建可寻址化资源也会创建出一些在构建应用时不会直接使用的文件：

  • addressables_content_state.bin：用于构建更新包，如果需要动态更新（热更？）之前构建过的内容，则需要存下每个旧的构建包对应的该文件。
  • AddressablesBuildTEP.json：记录构建性能数据

开始完整的内容构建

  1. 在 Build Settings 窗口中选择想要构建的平台 Platform Target
  2. 打开 Addressables Groups 窗口
  3. 通过Addressables Groups窗口中的构建按钮构建资源

构建流程开始

资源构建完成后就可以开始构建应用了，并且可以将之前设置的 RemoteBuildPath 路径下的文件上传到远程服务器上（CDN）

⚠重要：
如果你打算在不重新构建应用的情况下发布远程更新，你必须保留每一个已发布的构建的 addressables_content_state.bin 文件，如果没有这个文件的话，就只能重新构建完整的资源和应用了

远程资源分配 / 远程内容分发

你可以在 Addressables 的支持下通过内容分发网络 (CDN：Content Delivery Network) 或其他托管服务远程分发内容。Unity提供了Unity云（CCD: Cloud Content Delivery），当然也可以使用其他的CDN服务。

在构建远程分发内容之前，这些事情必须先搞定：

  • 勾选 Build Remote Catalog
  • 在我们发布内容所用的配置文件中配置 RemoteLoadPath ，用来映射远程URL。
  • 每个需要远程分发的资源包都需要将它的 Build Path 设置为 RemoteBuildPath，将 Load Path 设置为 RemoteLoadPath。
  • 在 Build Setting 窗口中设置想要打包导出的平台

最后要记得上传生成的资源文件（用于远程分发的资源文件）到CDN或其他远程服务器。

增量内容更新 Incremental content updates

增量更新可以让玩家下载更少的内容，不需要每次都整包重新下载，只需要下载更新的和增加的部分。如果要启用增量更新，则必须勾选上Build Remote Catalog，如果没有 remote catalog 的话，已经安装好的应用是不会检查是否有更新的。

开始进行一个增量更新

  1. 在 Build Settings 窗口里面将 Platform Target 设置成跟你想要更新的之前的那个包的相同的平台。
  2. 打开 Addressables Groups 窗口
  3. 在 Tools 的下拉菜单中选择 Check for Content Update Restrictions 命令
  4. 在打开的窗口中找到之前构建的addressables_content_state.bin文件，这个文件在Assets/AddressablesAssetsData 子文件夹（以平台命名）下
  5. 点击 Open，更新预览会查找更改并标记出来
  6. 点击应用更改
  7. 在 Build 菜单中国选择 Update a Previous Build 命令
  8. 打开之前构建所产生的addressables_contet_state.bin文件

构架过程开始

⚠注意：
Addressables是通过addressables_content_state文件来识别你的资源的更改的，每一次build所产生的这个文件都必须保存下来，否则下次你就只能更新整包而不能增量更新了。
